We implement thresholded attention, where a query only attends to a key if the q @ k score is above the threshold (this is implemented with the following idea, we have a max score (the max score any key has with the query), and we have a threshold tau, we only select keys, whose score with the query is >= max - tau. Because of the way we have implemented the threshold (ie with the max), we need to know the max score, hence I have 2 implementations, once where we use approx max score (ie max score seen till now, with this, we dont need 2 forward passes, we just do 1 forward pass, and end up selecting keys which are above the threshold), and the other one is exact max score (ie max score of the query with any key, for this we need 2 passes, in the first pass we solely determine the max score, and in the second we implement the threshold part). Even though the exact max score does 2 passes, which ends up making it slower, it gives a very large ppl boost at higher context lengths (approx max implementation has a ppl of 390.98 at 12k context length, while exact max implementation had a ppl of 12.2142 at 12k context length). We implement the threshold this way, as it ends up being more stable than an absolute threshold (ie just checking if q @ k > threshold)
And for now, I have implemented a shared thresholded strategy, ie once we have found out the keys which pass the threshold for a query, we do so for every query of the query tile, and we group together all the key sets, to get a pooled key set, then we rank the keys, based on their number of occurences across different key sets (corresponding to different queries of the query tile), and we end up picking top k% of the keys from the pooled set (I have chosen k to be 30).
And if at any moment the top k% of the keys exceeds a limit of 2048 tokens, we reduce our k so as to only include top 25%, or top 20%. However i have found that such an occurence (ie top 30% of pooled keys > 2048) occurs 1% of the times, and with the reduction to top 20% of pooled keys, should reduce that further
